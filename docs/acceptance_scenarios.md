# Приемо-сдаточные сценарии (UAT) — Sphinx

> Версия: 1.0  
> Дата: 2026-05-08  
> Продукт: Sphinx — AI-симулятор технического интервью

---

## Сценарий 1 — Регистрация / создание пользователя

**Цель:** Убедиться, что новый пользователь создаётся корректно.

**Предусловие:** Система запущена, БД доступна.

| Шаг | Действие | Ожидаемый результат |
|-----|----------|---------------------|
| 1 | Открыть главную страницу приложения | Страница загружается без ошибок, отображается форма запуска |
| 2 | POST `/users` c уникальным `telegram_id` | HTTP 200, возвращается объект с `id` и `telegram_id` |
| 3 | Повторить POST `/users` с тем же `telegram_id` | HTTP 200, возвращается тот же `id` (идемпотентность) |
| 4 | GET `/users/{id}` с полученным `id` | HTTP 200, поля совпадают |

**Критерий приёмки:** все шаги возвращают ожидаемые статусы без ошибок.

## Сценарий 2 — Запуск интервью

**Цель:** Подтвердить, что интервью запускается и первый вопрос возвращается.

| Шаг | Действие | Ожидаемый результат |
|-----|----------|---------------------|
| 1 | Выбрать уровень (junior/middle/senior) и стек | Кнопка «Начать» становится активной |
| 2 | POST `/interview/start` с валидными параметрами | HTTP 200, в ответе `interview_id` и `current_question` |
| 3 | Проверить поля вопроса | Присутствуют `id`, `text`, `topic`, `difficulty` |
| 4 | Запустить интервью с несуществующим `user_id` | HTTP 404 |

**Критерий приёмки:** первый вопрос генерируется без обращений к реальному LLM (mock в тестах).

---

## Сценарий 3 — Ответ на вопрос и получение оценки

**Цель:** Убедиться, что ответ обрабатывается и возвращается оценка.

| Шаг | Действие | Ожидаемый результат |
|-----|----------|---------------------|
| 1 | POST `/interview/{id}/answer` с текстом ответа | HTTP 200 |
| 2 | Проверить структуру `evaluation` | Поля: `score` (5 компонентов), `feedback`, `weak_topics` |
| 3 | Все компоненты `score` в диапазоне [0, 10] | Нет значений вне диапазона |
| 4 | POST `/interview/{id}/answer` повторно без нового вопроса | HTTP 409 |

**Критерий приёмки:** оценка возвращается в структурированном формате.

---

## Сценарий 4 — Получение результата интервью

**Цель:** Проверить финальный summary после прохождения всех вопросов.

| Шаг | Действие | Ожидаемый результат |
|-----|----------|---------------------|
| 1 | Ответить на ≥ 1 вопрос | HTTP 200 |
| 2 | GET `/interview/{id}/result` | HTTP 200 |
| 3 | Проверить поля ответа | `average_score`, `summary`, `questions_results` присутствуют |
| 4 | `questions_results` содержит все пройденные вопросы | Количество совпадает с кол-вом ответов |
| 5 | GET `/interview/99999/result` | HTTP 404 |

**Критерий приёмки:** средний балл вычислен корректно, список вопросов полный.

## Сценарий 5 — Адаптивный режим (v2)

**Цель:** Убедиться, что v2-эндпоинты работают и возвращают корректный формат.

| Шаг | Действие | Ожидаемый результат |
|-----|----------|---------------------|
| 1 | GET `/interview/{id}/question/v2` | HTTP 200, те же поля, что и v1 |
| 2 | POST `/interview/{id}/answer/v2` с ответом | HTTP 200, `evaluation` с теми же полями |
| 3 | GET `/interview/{id}/result/v2` | HTTP 200, идентичная структура result |

**Критерий приёмки:** v2 обратно совместим по структуре ответа с v1.

## Сценарий 6 — Сохранность данных при перезапуске

**Цель:** Данные интервью сохраняются после перезапуска backend-сервиса.

| Шаг | Действие | Ожидаемый результат |
|-----|----------|---------------------|
| 1 | Начать интервью, ответить на 2 вопроса | HTTP 200 |
| 2 | Перезапустить backend (`docker compose restart backend`) | Сервис поднимается |
| 3 | GET `/interview/{id}/result` | HTTP 200, данные сохранены в БД |

**Критерий приёмки:** история ответов не теряется при перезапуске контейнера.

## Сценарий 7 — Обработка недоступности LLM

**Цель:** Система деградирует gracefully при недоступности YandexGPT.

| Шаг | Действие | Ожидаемый результат |
|-----|----------|---------------------|
| 1 | Выставить заведомо неверный `YANDEX_API_KEY` | Ключ принят в конфиге |
| 2 | POST `/interview/start` | HTTP 503 с `detail` об ошибке LLM |
| 3 | Проверить логи | Запись уровня ERROR с причиной |

**Критерий приёмки:** нет HTTP 500 без тела, нет утечки stacktrace в ответе.

## Сценарий 8 — Проверка CORS

**Цель:** API принимает запросы только с разрешённого FRONTEND_URL.

| Шаг | Действие | Ожидаемый результат |
|-----|----------|---------------------|
| 1 | Браузерный запрос с `Origin: http://localhost:5173` | `Access-Control-Allow-Origin` содержит этот origin |
| 2 | Запрос с `Origin: http://evil.example.com` | Заголовок CORS отсутствует или не соответствует |

**Критерий приёмки:** другие origin блокируются браузером; серверный ответ не 500.

## Сценарий 9 — Healthcheck и мониторинг

**Цель:** Эндпоинты готовности и метрик отвечают корректно.

| Шаг | Действие | Ожидаемый результат |
|-----|----------|---------------------|
| 1 | GET `/health` | HTTP 200, `{"status": "ok"}` |
| 2 | GET `/metrics` (Prometheus) | HTTP 200, текст в формате Prometheus exposition |
| 3 | Остановить backend, сделать запрос | Alertmanager отправляет письмо в течение 2 минут |

**Критерий приёмки:** alert уходит на email при недоступности сервиса.

## Сценарий 10 — NPS / CSAT / CES метрики

**Цель:** Подтвердить сбор пользовательских метрик качества.

| Шаг | Действие | Ожидаемый результат |
|-----|----------|---------------------|
| 1 | POST `/feedback/nps` c `{"score": 9, "interview_id": <id>}` | HTTP 200 |
| 2 | POST `/feedback/csat` c `{"score": 4}` | HTTP 200 |
| 3 | POST `/feedback/ces` c `{"score": 2}` | HTTP 200 |
| 4 | GET `/metrics` | Счётчики `sphinx_nps_total`, `sphinx_csat_total`, `sphinx_ces_total` присутствуют |
| 5 | Grafana Dashboard «Executive» | Панели NPS/CSAT/CES отображают актуальные значения |

**Критерий приёмки:** метрики видны в Grafana в течение 1 минуты после отправки запроса.
